EVERY RELEASE
1. Freeze the RC identity and source commit.
2. Create a release review-request pack before requesting any review. It MUST
   contain role-specific Korean text files for Design_, Spec_, Impl_, Test_, and
   Devel_, plus an auxiliary-review assignment text file.
3. Instantiate the files from the six Design_Deeplus_Release_* templates. Each
   file MUST name the RC identity, source revision, changed inputs, questions,
   evidence required, report filename/prefix, deadline, and the requirement to
   present alternatives.
4. Save the pack as
   Design_Deeplus_<release>_Review_Request_Pack.zip, publish it with the RC,
   and record the required auxiliary roles and reasons in the manifest.
5. Publish the change/risk summary and extra-auditor assignment.
6. Require reports from Spec_, Impl_, Test_, and Devel_.
7. Require auxiliary reports triggered by syntax, security, concurrency, ABI,
   performance, package, documentation, or accessibility risk.
8. Reconcile conflicts by evidence and domain authority.
9. State separate verdicts for design, publication, implementation handoff,
   conformance, promotion, and product support.
10. Provide at least one viable alternative to disputed or deferred work.
11. Publish source, binary, documentation, and evidence artifacts separately.
12. Update the current pointer only after all release gates pass.

REPORT
Use an English Markdown filename beginning with Design_. Include:
- Release identity and evidence boundary.
- Lane verdicts.
- Required and received role reports.
- P0/P1/P2 and accepted risks.
- Integrated decisions and alternatives.
- Extra-auditor assignment and disposition.
- Path and digest of the role-specific review-request pack.
- Release manifest/provenance status.
- Next-release acceptance gates and handoff.

ON SCREEN

follow-up roles, and links to the report and ZIP.
